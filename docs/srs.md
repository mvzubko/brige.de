# Спецификация Требований к Программному Обеспечению (SRS)
## 1. Введение
### 1.1 Цель документа
Документ описывает требования к разработке формы для сервисных техников и команды продаж, предназначенной для:
- Фиксации деталей выполненных работ на объектах клиентов.
- Сбора изображений, дополнительной информации и электронной подписи клиента.
- Автоматической отправки отчётов на info@brige.de.
  
### 1.2 Область применения
Разрабатываемая форма будет использоваться на защищённом веб-сервисе, доступном для авторизованных пользователей (сервисных техников и менеджеров по продажам). Форма обеспечивает:
- Учет выполненных работ.
- Ведение базы данных по заявкам.
- Генерацию и рассылку отчетов.

### 1.3 Определения, сокращения и аббревиатуры
- IS – Информационная система.
- SRS – Спецификация Требований к Программному Обеспечению.
- GDPR – Общий регламент защиты данных.
- SSL – Протокол защиты соединений.

### 1.4 Ссылки на нормативные документы
- IEEE 830 – Стандарт для документации требований.
- GDPR – Регламент ЕС по защите персональных данных.
- W3C – Рекомендации по веб-стандартам.

### 1.5 Обзор документа
Документ SRS включает в себя описание общих характеристик системы, конкретных функциональных и нефункциональных требований, а также требований к интерфейсам, безопасности и развертыванию.

## 2. Общее описание системы
### 2.1 Перспектива продукта
Система является модулем внутри общей ИС предприятия, отвечающим за документирование и обработку данных выполненных работ. Интегрируется с электронной почтой для автоматической отправки отчетов, а также с системами хранения данных.

### 2.2 Основные функции продукта
- Регистрация заявок с генерацией уникального номера.
- Ввод данных о клиенте и сотруднике.
- Фиксация деталей выполненных работ и используемых материалов.
- Возможность загрузки медиафайлов (фотографии, документы).
- Сбор обратной связи и подписи клиента.
- Автоматическая отправка отчётов на указанный адрес.

### 2.3 Пользовательские характеристики
- Сервисные техники и представители отдела продаж.
- Пользователи с ограниченными правами доступа к форме.
- Администраторы ИС, ответственные за настройку и поддержку сервиса.

### 2.4 Ограничения системы
- Форма должна размещаться на защищённом поддомене (например, service.brige.de).
- Доступ только для авторизованных пользователей.
- Хранение данных и их передача должны соответствовать требованиям GDPR.
- Интеграция с почтовым сервером для отправки отчетов.

### 2.5 Предположения и зависимости
- Пользователи имеют стабильное интернет-соединение.
- Сервер обладает необходимыми сертификатами SSL.
- Операционные системы и браузеры у сотрудников соответствуют современным стандартам безопасности.

## 3. Специфические требования
### 3.1 Функциональные требования
#### 3.1.1 Регистрация и ввод данных
- Форма должна автоматически генерировать уникальный номер заявки.
- Реализовать поля для ввода:
  - Данных клиента (имя, компания, адрес, контактное лицо, номер телефона).
  - Данных техника или менеджера (имя, компания, дата визита, время прибытия/ухода).
  
#### 3.1.2 Выбор типа работ
- Реализовать набор флажков для выбора типа выполненной работы:
  - Установка, Ремонт, Обслуживание, Осмотр, Коммерческий визит, Консультация, Другое (с дополнительным текстовым полем).

#### 3.1.3 Описание работы и материалов
- Текстовое поле для подробного описания выполненных работ.
- Выпадающий список или мультивыбор для указания использованных материалов.
- Опция указать дополнительную работу:
  - Поле для описания и указания срочности (Низкая, Средняя, Высокая, Критическая).

#### 3.1.4 Загрузка медиафайлов
- Возможность множественной загрузки изображений.
- Загрузка дополнительных документов в форматах PDF, Word, Excel.

#### 3.1.5 Подтверждение и электронная подпись
- Поле для отметки успешного завершения работы (Да/Нет).
- Текстовое поле для комментариев клиента.
- Поле для ввода имени клиента и цифровой подписи.

#### 3.1.6 Отправка формы
- Наличие кнопки "Отправить", при нажатии на которую:
  - Форма отправляет данные, изображения и подпись на info@brige.de.
  - Опционально формируется PDF-отчет для архивирования.

### 3.2 Нефункциональные требования
#### 3.2.1 Безопасность
- Форма должна размещаться на защищённом поддомене с SSL-шифрованием.
- Доступ к форме должен быть с ограничением по паролю, только для авторизованных пользователей.
- Все хранимые данные должны соответствовать регламенту GDPR.

#### 3.2.2 Производительность
- Время отклика формы не должно превышать установленные нормативы (например, 3 секунды).
- Система должна поддерживать одновременную работу нескольких пользователей.

#### 3.2.3 Удобство использования
- Интуитивно понятный интерфейс для быстрого ввода данных.
- Адаптивный дизайн для работы на мобильных устройствах и планшетах.

#### 3.2.4 Надёжность
- Резервное копирование данных и возможность их восстановления в случае системного сбоя.
- Автоматическое логирование событий и ошибок.

### 3.3 Требования к интерфейсам
#### 3.3.1 Пользовательский интерфейс
- Современный, чистый и удобный дизайн.
- Поддержка многоязычности (при необходимости).

#### 3.3.2 Интерфейсы обмена данными
- Интеграция с почтовыми системами для автоматической отправки отчётов.
- API для передачи данных в центральную систему управления заявками (при наличии).

## 4. Системные и аппаратные требования
- Сервер с установленным SSL-сертификатом.
- Совместимость с основными современными веб-браузерами.
- Поддержка мобильных устройств (смартфоны и планшеты).
